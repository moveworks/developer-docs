# Description

This plugin enables admins and analysts to query and analyze **Moveworks platform usage data** directly through the **Moveworks AI Assistant**, eliminating the need to navigate dashboards or export raw data manually.

Understanding how employees interact with the AI Assistant — which plugins get used, which knowledge articles get cited, how conversations route across domains — typically requires exporting data, building custom reports, or waiting on BI teams. This plugin removes that friction by letting users ask analytical questions in natural language and get instant answers backed by real platform data.

# Pre-requisites

Before installing and using the **DataAPI Unified Analytics** plugin, please ensure the following requirements are met:

## **1. Moveworks Data API Connector**

This plugin requires an active **Data API connector** (preferably API key) to communicate with the Moveworks Data API.

- If you have not already configured the connector, please follow the [**Data API — Connector Guide**](https://marketplace.moveworks.com/connectors/moveworks-data-api#how-to-implement) to set up an API key or OAuth token for your org.
- For full details on Data API integration and available objects, refer to the [**Data API — How to Build Integration with the API**](https://help.moveworks.com/ai-assistant/analytics-performance/data-api/how-to-build-integration-with-the-api) documentation.
- The connector must be fully set up before installing this plugin.
- Once the connector is successfully configured, follow our [**plugin installation documentation**](https://help.moveworks.com/docs/ai-agent-marketplace-installation) for detailed steps on how to install and activate the plugin in **Agent Studio**.

> **Note**: The plugin does not write or modify any data. It is strictly read-only and uses GET requests across all endpoints.

## API Details

### **API #1: Get Interactions Count**

This API fetches the total number of interactions within a date range. Used in both the fast path (to return the count directly) and the deep path (to calculate pagination offsets).

```bash
curl --request GET \
  --url 'https://api.moveworks.ai/export/v1/records/interactions/$count?$orderby=created_time&$filter=created_time gt '{{start_date}}T00:00:00Z' and created_time lt '{{end_date}}T00:00:00Z'' \
  --header 'Authorization: Bearer <API_KEY>' \
  --header 'Accept: application/json'
```

**Query Parameters:**

- `start_date` *(string):* Start of the date range in `YYYY-MM-DD` format.
- `end_date` *(string):* End of the date range in `YYYY-MM-DD` format.

**Returns:** Integer count.

---

### **API #2: Get Raw Interactions Data**

This API fetches paginated raw interaction records for deep analysis. The compound action calculates offsets based on the count and fetches all pages in parallel.

```bash
curl --request GET \
  --url 'https://api.moveworks.ai/export/v1/records/interactions?$skip={{skip}}&$orderby=created_time&$filter=created_time gt '{{start_date}}T00:00:00Z' and created_time lt '{{end_date}}T00:00:00Z'' \
  --header 'Authorization: Bearer <API_KEY>' \
  --header 'Accept: application/json'
```

**Query Parameters:**

- `skip` *(string):* Pagination offset (increments of 500).
- `start_date` *(string):* Start of the date range.
- `end_date` *(string):* End of the date range.

**Key Fields Returned (per interaction):**

- `actor` *(string):* `user` or `bot` — use to separate user inputs from AI responses.
- `type` *(string):* Action type: `FREE_TEXT`, `BUTTON_CLICK`, `LINK_CLICK`, `BOT_MESSAGE`.
- `detail.domain` *(string):* The detected domain (IT, HR, FINANCE, etc.). Only populated for user free-text.
- `detail.entity` *(string | null):* Detected search topics (e.g., `vpn`, `pto`).
- `detail.content` *(string | null):* The raw content of the interaction.
- `detail.resource_id` *(string | null):* Resource ID if the interaction referenced a specific resource.
- `platform` *(string):* Interaction surface: `msteams`, `slack`, `web`.
- `conversation_id` *(string):* Links to the parent conversation.
- `user_id` *(string):* Links to the user record.
- `created_time` *(string):* Timestamp of the interaction.
- `parent_interaction_id` *(string | null):* The parent interaction, if this is a follow-up.
- `label` *(string | null):* Classification label applied to the interaction.

**Key Nuances:**

- The `detail.domain` and `detail.entity` fields are only populated for interactions where `actor` is `user` and `type` is `FREE_TEXT`. Bot messages and button clicks do not carry domain/entity annotations.
- Use the `platform` field to analyze interaction surface distribution (e.g., Slack vs Teams vs Web adoption).

---

### **API #3: Get Users Count**

This API fetches the total number of ingested users updated since yesterday (dynamic date).

```bash
curl --request GET \
  --url 'https://api.moveworks.ai/export/v1/records/users/$count?$orderby=id&$filter=last_updated_time gt '{{yesterday_date}}T06:27:58.321Z'' \
  --header 'Authorization: Bearer <API_KEY>' \
  --header 'Accept: application/json'
```

**Query Parameters:**

- `yesterday_date` *(string):* Yesterday's date in `YYYY-MM-DD` format. Automatically calculated within the compound action.

**Returns:** Integer count.

# **What Is In Scope for This Plugin?**

This plugin supports the following capabilities:

- Query **total counts and volumes** for any of the five datasets — Interactions, Users, Conversations, Plugin Calls, and Plugin Resources — with near-instant results via the fast path.
- Run **deep analytics** using the Code Interpreter (SDA), including week-over-week trends, cross-dataset joins, statistical aggregations, and distribution breakdowns.
- Analyze **interaction patterns**: trending search topics, interaction surface distribution (Slack vs Teams vs Web), domain distribution (IT, HR, Finance), and actor breakdowns (user vs bot).
- Analyze **user demographics**: bot access status, language preferences, external identity integration coverage (Okta, ServiceNow, Slack), and user tenure.
- Analyze **conversation routing**: volume by domain, routing patterns (DM vs Notification vs Ticket Interception), and daily conversation trends.
- Analyze **plugin performance**: top plugins by usage volume, served vs used rates, native vs custom plugin adoption, and plugin latency.
- Analyze **knowledge effectiveness**: citation rates, top cited resources, resource type breakdowns (Knowledge vs Forms), and snippet-level transparency.
- Query data using **natural language date expressions**, such as "last week", "this month", "last 30 days", or specific date ranges.
- Perform **cross-dataset analysis** by joining on shared keys (`user_id`, `conversation_id`, `id`).
- Handle date ranges **up to 30 days** for deep analysis, aligning with the Data API's TTL window. Simple count queries return near-instantly within this range.

# **What Is Out of Scope for This Plugin?**

This plugin does **not** support the following:

- **Querying data older than 30 days** — the Moveworks Data API is implementing a **30-day TTL (time-to-live)**, meaning the API will only allow data ingestion up to 30 days back. Queries requesting data beyond this window will not return results. For more details, see the [**Proactive Insights — Data API Community Post**](https://community.moveworks.com/stakeholder-tools-exi-mw-setup-ks-analytics-115/big-news-proactive-insights-are-coming-to-the-data-api-3286).
- **Exporting raw data** — this is an on-demand analytics tool, not a data export or ETL pipeline. Users cannot ask to "export all historical data."
- **Replacing a BI dashboard** — the plugin is designed for scoped, analytical questions. It should not be treated as a replacement for an organization's full BI or reporting infrastructure.
- **Writing or modifying data** — all operations are strictly read-only.
- **Querying all 5 datasets simultaneously with very large record counts** — there is a hard payload limit of **100MB raw**, and the reasoning engine's working memory caps at **~400KB compressed**. Fetching tens of thousands of records across all 5 objects in a single request can cause failures.
- **Queries that exceed the 2-minute execution timeout** — Agent Studio enforces a strict **160-second total lifecycle timeout**. Running up to five parallel pagination loops concurrently increases the risk of timing out, especially on queries spanning the full 30-day window across multiple datasets.
- Providing **recommendations, advice, or best practices** on how to improve metrics — the plugin strictly reports data and analytics. For optimization guidance, use Knowledge Search plugins.
- **Real-time or streaming data** — the plugin queries point-in-time snapshots from the Data API export endpoints.
- Querying data **on behalf of another organization** or across tenant boundaries.
- **Handling ambiguous or very broad requests with full precision** — because this plugin consolidates 5 datasets into a single "catch-all" description, the reasoning engine's intent routing can occasionally be less deterministic for vague queries compared to 5 individually scoped plugins.
